Software Testing

What is Software Testing?

  • It is not sufficient to demonstrate that the software is doing what it is supposed to do.

  • It is additional necessary to demonstrate that the computer code isn't doing what's not supposed to do.

  • Software testing can also be stated as the process of validating and verifying that a software program / application / product:

  • Works obviously in line with the business and technical requirement

  • Works consistently and predictably.

  • Process of finding defects i.e. variance between Expected results and Actual results.

  • Process of executing a software program application with intent of finding errors.

Why is testing necessary?

  • Software testing is important to make sure the product or application is defect free, as per customer specifications.

  • Software testing identifies faults whose removal increases the software quality and increases the software’s trustworthiness.

  • Testing effort is directly proportional to the complexity of the program.

  • More complexity the program, more the testing effort is required.

  • Software should be:

    • Error Free

    • Efficient

    • Secured

  • Software testing is important as it may cause mission failure, impact on operational performance and unreliable if not done right.

Start testing-when?

  • Testing starts right from the requirements phase and continues until the release time.

  • Objective of starting early:

  • Requirements related defects caught later in the SDLC result in higher cost to fix the defect.

Stop testing-when?

    Following factors are considered:-

  • Test cases executed with acceptable percentage of defects

  • Test budget has run out

  • Project Deadlines e.g. release deadlines, testing deadlines

  • Coverage of code, functionality or requirements reaches to specific point

  • Bug rate falls below acceptable level.

Participants in testing

  • Customer

  • User

  • Developer

  • Tester

  • Auditor

Common problems in the software development process?

  • Poor Requirements – unclear, incomplete, too general

  • Unrealistic Schedule – If too much work is crammed in too little time, problems are inevitable

  • Inadequate testing – No one will know whether or not the program is any good till the customer complains or system crashes.

  • Features – A request to pile on new features after development is underway; extremely common.

  • Miscommunications – If developers don’t know what’s needed or customers have erroneous expectations, problems are guaranteed.

Best practices in testing

  • Test planning

  • Code Testability

  • Test early

  • Measure test costs, coverage, results and effectiveness.

  • Negative Testing : Needs kick the wall approach

Misconceptions about testing

  • Anyone can test software: No particular skill is required.

  • Testers can test the quality at the end of the project!

  • Defects found means blaming to the developers.

Testing Principles

  • 1. Testing is context dependent

    Testing is done differently in different context

    For Eg:- Safety critical software Is tested differently from e-commerce site

  • 2. Exhaustive testing is impossible

    Testing everything is not feasible except trivial cases. Instead of exhaustive testing we use risks and priorities to focus testing efforts.

  • 3. Early testing

    Testing activities should start as early as possible in software or SDLC and should focused on defined objectives.

  • 3. Early testing

    Testing activities should start as early as possible in software or SDLC and should focused on defined objectives.

  • 4. Defect Clustering:-

    A small module holds most of the defects discovered during pre-release testing or show the maximum operational failure

  • 5. Pesticide paradox

    If the same tests are repeated over and over again, eventually the same set of test cases will no longer find way new bugs. To overcome this problem, the test case needs to be regularly reviewed and revised to potentially find more defects.

  • 6. Testing shows presence of defects

    Testing can show that defects are present but cannot prove that there are no defects

  • 7. Absence of error fallacy

    Finding and fixing defects not help if the system built is unusable and does not satisfy the user’s expectations.
